## 1. Executive Summary

Academic institutions require students and researchers to produce structured book reports that synthesize literary themes, critical arguments, historical context, and academic citations into a single, formatted document. This process is manual, time-consuming, and prone to formatting errors when working directly with typesetting environments such as LaTeX.

This project defines and demonstrates a **hierarchical multi-agent AI pipeline** that automates this workflow. The system receives a user's query about a target book, decomposes the generation task into specialized subtasks, and produces a publication-ready LaTeX/PDF academic report through orchestrated agent cooperation and autonomous quality assurance.

The primary deliverable of this proof-of-concept is a **58-page compiled PDF academic paper** that:
1. Formally specifies the proposed multi-agent architecture.
2. Provides mathematical frameworks for evaluating agent performance and resource allocation.
3. Demonstrates the system through four empirical case studies (Piketty, Turing, Kuhn, Machiavelli).
4. Documents the development process, lessons learned, and system limitations.

## 2. Problem Statement

### 2.1 The Core Problem

| Problem | Description |
|---------|-------------|
| Manual synthesis bottleneck | Academic book reports require prolonged manual synthesis of extensive texts. |
| Monolithic LLM limitations | Single-prompt LLM instances fail to sustain formatting constraints over lengthy manuscripts, causing layout drift, missing citations, and compiler exceptions. |
| Compilation errors | LaTeX documents generated by AI frequently contain unescaped special characters, overfull `\hbox` violations, missing delimiters, and broken bibliography references. |
| Context window exhaustion | Processing multi-chapter books in a single context window leads to context dilution and semantic drift. |
| Token budget constraints | Systems operate under hard token ceilings (18,000,000 tokens in this system), requiring explicit resource optimization. |

### 2.2 Root Cause

Monolithic LLMs balance multi-layered semantic constraints simultaneously — narrative generation, citation management, table layout, mathematical typesetting — in a single context. This creates processing conflicts that manifest as structural drift, broken syntax, and compilation failures in long documents.

## 5. User Stories

### 5.1 Core User Flow

**US-01:** As a student, I want to provide the name of a book and a topic prompt, so that the system generates a structured LaTeX academic report without me writing any LaTeX manually.

**US-02:** As a student, I want the generated report to compile without errors, so that I receive a valid PDF on the first or second LuaLaTeX invocation.

**US-03:** As a student, I want citations to be correctly formatted and consistent throughout the document, so that the bibliography is academically valid.

**US-04:** As a student, I want tables, figures, and equations to be properly numbered and cross-referenced, so that the table of contents and in-text references resolve correctly.

### 5.2 System User Stories

**US-05:** As the Orchestrator Agent, I want to receive a structured user query and delegate subtasks to specialized agents, so that each agent focuses only on its domain.

**US-06:** As the Text Agent, I want to generate long-form academic prose using Chain-of-Thought prompting and RAG-retrieved style templates, so that I produce consistent, scholarly narrative without thematic hallucinations.

**US-07:** As the Table Agent, I want to compute column widths dynamically before generating tabular environments, so that no `\hbox` overflow warnings occur.

**US-08:** As the Citation Agent, I want to validate every in-text citation against a reference registry, so that no undefined citation keys reach the LuaLaTeX compiler.

**US-09:** As the QA Agent, I want to intercept LuaLaTeX compilation logs and dispatch targeted fixes to the source `.tex` file, so that the document compiles successfully without human debugging.

## 6. Functional Requirements

### 6.1 Orchestration Layer

| ID | Requirement |
|----|-------------|
| F-01 | The Orchestrator Agent shall parse incoming user queries using morphological marker classification to activate the appropriate specialist agents. |
| F-02 | The Orchestrator shall maintain a transient state ledger receiving structured JSON payloads from parallel sub-agents. All inter-agent communication uses direct JSON payloads (not the Model Context Protocol — see `ADR.md §ADR-004` for rationale). |
| F-03 | The Orchestrator shall enforce a 18,000,000 token ceiling using a greedy CER-based content selection algorithm. |
| F-04 | The Orchestrator shall coordinate a Raft-based consensus protocol (§3.5) when parallel agents detect conflicting layout parameters. Note: this is an architectural design pattern described in §3.5; it is not fully deployed in the proof-of-concept implementation. |

### 6.2 Text Generation

| ID | Requirement |
|----|-------------|
| F-05 | The Text Agent shall generate academic prose using Chain-of-Thought (CoT) prompting with domain-specific few-shot examples. |
| F-06 | The Text Agent shall prevent stylistic drift by loading historical context vectors from the external vector database before each chapter synthesis. |
| F-07 | The Text Agent shall filter all output through a syntax verification matrix before writing to disk. |

### 6.3 Table Generation

| ID | Requirement |
|----|-------------|
| F-08 | The Table Agent shall compute column width allocations using the formula W_cell(c) = W_total × (γ_c / Σγ_k) before generating any tabular environment. |
| F-09 | The Table Agent shall use the booktabs standard (`\toprule`, `\midrule`, `\bottomrule`) for standard data tables. Bordered comparison tables (Tables 2 and 4 in the deliverable) use `\hline` rules by design to provide visual emphasis appropriate for architectural and transition-matrix layouts. |
| F-10 | The Table Agent shall ensure no `Overfull \hbox` warning causes visible page overflow. Wide tables shall be wrapped with `\resizebox{\textwidth}{!}{...}` or `\makebox[\textwidth][c]{...}` to contain content within the text area. |

### 6.4 Citation Management

| ID | Requirement |
|----|-------------|
| F-11 | The Citation Agent shall maintain a central reference registry and validate every `\cite{}` key before document assembly. |
| F-12 | The Citation Agent shall automatically replace raw unverified reference indicators with standardized `\cite{Key}` commands via regex substitution. |
| F-13 | The Citation Agent shall conform to IEEE citation style conventions. |

### 6.5 Diagram Generation

> **Scope note:** The Visualization Agent (§6.3 of the paper) functions as a diagram production subsystem. It is not represented in the primary orchestration diagram (Figure 1) or agent roles table (Table 1) of the paper, which both show the simplified primary flow. F-14 and F-15 describe this subsystem's requirements.

| ID | Requirement |
|----|-------------|
| F-14 | The Visualization Agent shall generate all figures using native TikZ/pgfplots commands, avoiding external raster images. |
| F-15 | Generated diagrams shall define nodes, arrows, and labels from abstract topology metadata, not hardcoded pixel coordinates. |

### 6.6 Quality Assurance

| ID | Requirement |
|----|-------------|
| F-16 | The QA Agent shall intercept raw LuaLaTeX compilation logs after every compilation pass. |
| F-17 | The QA Agent shall dispatch the `qa-repair-agent` to the source `.tex` file — never to the compiled PDF — when errors are detected. |
| F-18 | The system shall resolve all four error classes: `Overfull \hbox`, missing delimiter, BiDi alignment shift, and unresolved reference. |
| F-19 | The self-healing loop shall be deterministic: the same input error shall always trigger the same remediation protocol. |

### 6.7 Document Compilation

| ID | Requirement |
|----|-------------|
| F-20 | The system shall invoke LuaLaTeX twice per compilation cycle (`--interaction=nonstopmode`) to resolve all cross-references. |
| F-21 | The final PDF shall include a table of contents, numbered sections, numbered figures, numbered tables, and a references section. |
| F-22 | Hyperlinks (section, citation, URL) shall be rendered in blue using the hyperref package. |

## 8. Mathematical Optimization Requirements

The Orchestrator Agent must implement the following formal optimization framework:

**Content Selection (Greedy CER):**
- Rank content modules by CER_i = TokenCost_i / B_i (ascending)
- Select modules greedily until token ceiling is reached
- Demonstrated optimal: {Philosophical Synthesis (3.0M), Critical Character Analysis (4.0M), Thematic Summary Module (6.0M)} = 13.0M tokens, B = 228.5

**Evaluation Scoring (MCDA):**
- B_i = Σ W_j · S_ij where W = {Thematic Depth: 0.35, Stylistic Complexity: 0.25, Syntactic Feasibility: 0.20, Token Efficiency: 0.20}
- Normalize raw scores to [0,100] via Min-Max before applying weights

## 10. Out of Scope

The following are explicitly outside the scope of this proof-of-concept:

- Real-time web scraping of book content
- Integration with external APIs (Google Scholar, Semantic Scholar)
- User interface beyond the CLI
- Production deployment on cloud infrastructure
- Multi-user concurrency (single-session proof-of-concept only)
- Automated bibliography file (`.bib`) generation via BibTeX — references are managed inline
